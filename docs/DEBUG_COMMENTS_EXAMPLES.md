# Debug Comments - Live Examples

This document shows actual examples of debug comments generated by the system.

## CSS Debug Comment Example

When debug comments are enabled (development mode), CSS resources are wrapped like this:

```css
/*
 * ==============================================================================
 * EMBEDDED RESOURCE DEBUG INFO
 * ==============================================================================
 * generated_at: 2026-01-12T13:26:40.877068
 * source_file: assets/css/style.css
 * size_bytes: 45678
 * size_kb: 44.61
 * type: css
 * description: Main application styles
 * ==============================================================================
*/
body {
  background-color: #0d1117;
  color: #c9d1d9;
  font-family: 'Roboto', sans-serif;
}
/* ... more CSS ... */
/* END OF CSS: assets/css/style.css */
```

## JavaScript Debug Comment Example

JavaScript modules are wrapped with similar metadata:

```javascript
/*
 * ==============================================================================
 * EMBEDDED RESOURCE DEBUG INFO
 * ==============================================================================
 * generated_at: 2026-01-12T13:26:40.877159
 * source_file: assets/js/*.js (modular)
 * size_bytes: 123456
 * size_kb: 120.56
 * type: js
 * description: Modular application code (concatenated)
 * modules: storage.js, filters.js, map.js, speech-bubbles.js, utils.js, template-engine.js, dropdown.js, dashboard-ui.js, filter-description-ui.js, event-listeners.js, app.js
 * ==============================================================================
*/

// ============================================================================
// MODULE: storage.js
// SOURCE: assets/js/storage.js
// ============================================================================
const StorageManager = {
  save(key, data) {
    // ... storage code ...
  }
};

// ============================================================================
// MODULE: app.js
// SOURCE: assets/js/app.js
// ============================================================================
class App {
  init() {
    // ... app initialization ...
  }
}
/* END OF JS: assets/js/*.js (modular) */
```

## HTML Component Boundary Markers

HTML components are wrapped with visual boundary markers:

```html
<!-- ▼ START COMPONENT: assets/html/html-head.html ▼ -->
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>KRWL> Events from here til sunrise</title>
  <!-- ... more head content ... -->
</head>
<!-- ▲ END COMPONENT: assets/html/html-head.html ▲ -->

<!-- ▼ START COMPONENT: assets/html/map-main.html ▼ -->
<div id="map"></div>
<!-- ▲ END COMPONENT: assets/html/map-main.html ▲ -->

<!-- ▼ START COMPONENT: assets/html/dashboard-aside.html ▼ -->
<aside class="dashboard">
  <!-- ... dashboard content ... -->
</aside>
<!-- ▲ END COMPONENT: assets/html/dashboard-aside.html ▲ -->

<!-- ▼ START COMPONENT: assets/html/filter-nav.html ▼ -->
<nav class="filter-bar">
  <!-- ... filter controls ... -->
</nav>
<!-- ▲ END COMPONENT: assets/html/filter-nav.html ▲ -->
```

## Benefits of These Comments

### For Debugging
- **Quick Source Identification**: Immediately see which file contains specific code
- **Size Analysis**: Identify large resources that may need optimization
- **Timestamp Verification**: Confirm HTML is freshly generated
- **Component Structure**: Understand how templates are assembled

### For Development
- **No Guesswork**: No need to search through source files
- **Better Context**: Understand the build process
- **Easy Navigation**: Jump directly to source files
- **Performance Insights**: See actual resource sizes

### For Maintenance
- **Clear Attribution**: Know where each piece of code originates
- **Update Tracking**: Timestamps help track when changes were applied
- **Module Organization**: See how modular JS files are combined
- **Template Flow**: Understand component hierarchy

## When Debug Comments Appear

### ✅ Enabled (Development Mode)
- Running locally on developer's machine
- Environment variable `ENVIRONMENT` not set to `production`
- Not running in CI environment

### ❌ Disabled (Production/CI Mode)
- Deployed to Vercel, Netlify, Heroku, etc.
- Running in GitHub Actions, GitLab CI, etc.
- Environment variable `ENVIRONMENT=production` or `NODE_ENV=production`

## Viewing Debug Comments

### In Generated HTML
```bash
# Search for debug comments
grep -c "EMBEDDED RESOURCE DEBUG INFO" public/index.html

# View CSS debug section
grep -A 20 "EMBEDDED RESOURCE DEBUG INFO" public/index.html | grep -A 20 "type: css"

# View component boundaries
grep "START COMPONENT\|END COMPONENT" public/index.html
```

### In Browser DevTools
1. Open DevTools (F12)
2. Go to Sources tab
3. Open `index.html`
4. Search for "EMBEDDED RESOURCE DEBUG INFO"
5. See debug comments inline with code

## File Size Impact

Debug comments add approximately:
- **CSS**: ~300 bytes per resource (header + footer)
- **JavaScript**: ~400 bytes per resource
- **JSON**: ~200-500 bytes per data object (with pretty-printing)
- **HTML**: ~100 bytes per component boundary

**Total overhead**: ~5-10 KB in development mode (negligible for debugging)

This overhead is automatically removed in production builds!
